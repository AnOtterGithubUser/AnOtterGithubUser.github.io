Alphacross, part 1

June 17th, 2020

Introduction

Since October 2019 I have been working on a side-project with a friend. The aim is to build a web-app on which visitors will be able to play board games against an AI. The idea was to work on Reinforcement Learning and improve our coding skills.
We agreed on a few ground rules:
    - We should start on the lowest possible level. That means no high-level libraries like OpenAI Gym or RLCoach. We would code our own environments and use a standard Deep Learning library. In our case we set on Tensorflow.
    - We would focus not only on building a software that works but a well-written software. That means testing, continuous integration and deployment, clear function names and logic, and more testing.

Building the environment

The first environment would be the classic Connect4 game. The main challenges were to represent the grid and to assess a player's victory. For the first, the grid is represented as a simple list of lists representing the columns. This is way easier for inserting new tokens. The second was a bit more challenging and it was pretty easy to forgot edge cases.
However that is when testing intervene. As an afficionado of Software Craftmanship, I developped most of the Connect4 features using Test Driven Development. A simple game like this one is actually a great TDD exercise.
But Connect4 is not supposed to be the only environment in AlphaCross. Hence I wondered how we could separate the general functions that all future environments will need and the ones specific to Connect4. The answer was to implement an abstract parent class _Environment. The idea is that any other class will only interact with an environment through the general _Environment methods. In C++, we would simply set all other methods as private. However this does not exist in Python and the general consensus to precede private methods with an "_" did not seem strong enough a constraint.
